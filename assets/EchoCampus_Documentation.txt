EchoCampus — Product & Architecture Reference
==============================================

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. MODULES & BUSINESS RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1.1  Announcements
     - Only faculty users can create announcements.
     - The author_id field references public.directory (not public.users directly).
       The AnnouncementList component performs a JOIN on this relationship.
     - All authenticated users (students, faculty, admin) can read announcements.
     - A faculty user must have a row in public.faculty_users to post; their
       faculty_users.faculty_id must match the announcement's author_id.

1.2  Complaint Box
     - Only students can submit complaints.
     - is_anonymous (boolean, default true) must be explicitly passed in the
       insert payload — do not assume a default on the client side.
     - Upvotes are stored separately in public.complaint_upvotes.
       One row per (complaint_id, user_id) pair — enforced by a UNIQUE constraint.
     - A student can add an upvote (INSERT) or remove their own upvote (DELETE).
       There is no UPDATE path.
     - Limit: 1 complaint per student per 7 days (enforced by DB trigger).

1.3  Marketplace
     - Only students can create listings and view listings.
       Faculty and admin do not have read access to public.marketplace via RLS.
     - owner_email is a stored field — it is not derived from auth at read time.
       It must be included in the insert payload and kept in sync with the user's email.
     - price must be between 1 and 99,999 (numeric, DB constraint).
     - contact_info must be exactly 10 digits (regex enforced at DB level).
     - Only the listing owner can update a listing (e.g., mark is_sold = true).
     - Limit: 1 listing per student per 3 days (enforced by DB trigger).

1.4  Lost & Found
     - Any authenticated user can submit a lost/found report (students, faculty, admin).
     - Fields: title, location_found, contact_info are required.
             description and image_url are optional.
             is_resolved defaults to false.
     - Only the submitting user can delete their own post.
     - Limit: 2 posts per user per 24 hours (enforced by DB trigger).

1.5  Faculty Directory
     - public.directory is the authoritative faculty record store.
     - It is separate from public.faculty_profiles (which holds extended profile data
       linked to a logged-in user's account).
     - All authenticated users can read directory records.
     - No client-side write policy exists — directory records are managed externally
       (e.g., seeded by admin or via Supabase dashboard).
     - Faculty identity at signup is determined by email match against this table.

1.6  Anonymous Chat
     - Powered entirely by Firebase (Anonymous Auth + Firestore).
     - Each student has a session_code in public.student_profiles, which is used
       to generate a display label in chat without exposing identity.
     - No Supabase interaction occurs during chat; the two systems are independent.

2. ACCESS CONTROL MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Role Assignment
     - Role is stored in public.users.role.
       Allowed values: 'student' | 'faculty' | 'admin'
     - Assigned automatically at signup via the on_auth_user_created trigger:
         → Email found in public.directory  →  role = 'faculty'
         → Email not found                  →  role = 'student'
     - 'admin' cannot be self-assigned. It must be set directly in the database.
     - If a user already has role = 'admin', the trigger will not overwrite it on
       subsequent logins (protected by the ON CONFLICT update logic).
     - There is intentionally no client-accessible UPDATE policy on public.users.
       Role changes must go through an RPC or direct DB access to prevent privilege
       escalation.

2.2  Route Guards (App Layer)
     Enforcement happens in three places simultaneously:
       - src/middleware.ts              →  server-side route guard
       - src/components/ProtectedRoute.tsx  →  client-side guard
       - app/auth/login/page.tsx        →  post-login routing logic

     Route prefix access:
       /main/student/*   →  role = 'student' only
       /main/faculty/*   →  role = 'faculty' or 'admin'
       /main/*           →  unauthenticated → redirect to /auth/login

2.3  RLS Policy Matrix

     Table                 SELECT                 INSERT                      UPDATE          DELETE
     ─────────────────────────────────────────────────────────────────────────────────────────────────
     users                 Own row only           Self (email + role check)   None (blocked)  —
     student_profiles      All authenticated      Own row (user_id = uid)     Own row         —
     faculty_profiles      Own row only           Own row (faculty/admin)     Own row         —
     faculty_users         Own row only           Self-mapped to directory    —               —
     directory             All authenticated      None (client blocked)       —               —
     announcements         All authenticated      faculty_users match req'd   —               —
     complaint_box         All authenticated      Students only               —               —
     complaint_upvotes     All authenticated      Students only               —               Own row
     marketplace           Students only          Students only (own owner)   Owner only      —
     lost_found            All authenticated      Any authenticated           —               Own row

2.4  Helper Function
     public.is_faculty_email(input_email text) → boolean
     - Checks if an email exists in public.directory.
     - SECURITY DEFINER: executes with elevated rights; search_path locked to public.
     - Granted to: anon, authenticated.
     - Used during signup to determine role assignment.

3. DATABASE SCHEMA OVERVIEW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Core tables and their primary relationships:

  auth.users  (Supabase managed)
       │
       └──▶  public.users  (id, email, full_name, role, created_at)
                  │
                  ├──▶  public.student_profiles  (user_id, session_code)
                  │
                  ├──▶  public.faculty_profiles  (user_id, department, phone_no,
                  │                               cabin_no, experience_years)
                  │
                  ├──▶  public.faculty_users  (user_id, faculty_id)
                  │              └────────────────────────────┐
                  │                                           ▼
                  │                               public.directory  (id, name, email,
                  │                               department, phone_no, date_of_birth,
                  │                               cabin_no, experience)
                  │                                           │
                  │                                           └──▶  public.announcements
                  │                                                  (author_id → directory.id)
                  │
                  ├──▶  public.complaint_box  (user_id, content, is_anonymous)
                  │              │
                  │              └──▶  public.complaint_upvotes  (complaint_id, user_id)
                  │                    [UNIQUE on complaint_id + user_id]
                  │
                  ├──▶  public.marketplace  (owner_id, owner_name, owner_email,
                  │                         product_title, description, price,
                  │                         contact_info, is_sold)
                  │
                  └──▶  public.lost_found  (user_id, title, description,
                                           location_found, contact_info,
                                           image_url, is_resolved)

Key constraint notes:
  - users.id is a FK to auth.users(id) with ON DELETE CASCADE.
  - announcements.author_id references directory(id) with ON DELETE RESTRICT
    (an announcement cannot be deleted by removing a directory entry).
  - complaint_upvotes has a composite UNIQUE(complaint_id, user_id) — prevents
    duplicate upvotes without needing application-layer checks.
  - marketplace.price: numeric, must be > 0 and <= 99999.
  - marketplace.contact_info: must match regex '^[0-9]{10}$'.
  - faculty_profiles.experience_years and directory.experience: must be >= 0.

4. TRIGGERS & FUNCTIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  Usage Limit Triggers  (BEFORE INSERT, per row)

     Trigger                      Table              Function                      Limit
     ─────────────────────────────────────────────────────────────────────────────────────
     enforce_lost_found_limit     lost_found         check_lost_found_limit()      2 posts / 24 hrs
     enforce_complaint_limit      complaint_box      check_complaint_limit()       1 post  / 7 days
     enforce_marketplace_limit    marketplace        check_marketplace_limit()     1 listing / 3 days

     All three raise a user-readable exception on limit breach.
     The exception message is surfaced to the client by Supabase and can be caught
     in the API route handler to return a 429-style error response.

4.2  Auth Sync Triggers

     Trigger                       Event                         Function
     ────────────────────────────────────────────────────────────────────────────────────
     on_auth_user_created          AFTER INSERT on auth.users    handle_new_auth_user()
     on_auth_user_email_updated    AFTER UPDATE(email)           handle_auth_user_email_update()
                                   on auth.users

     handle_new_auth_user() logic:
       1. Looks up the new user's email in public.directory.
       2. Inserts into public.users with role = 'faculty' if matched, 'student' otherwise.
          - ON CONFLICT: updates email and full_name; preserves 'admin' role if already set.
       3. If directory match found:
          - Upserts into public.faculty_profiles (department, phone_no, cabin_no,
            experience_years pulled from directory).
          - Upserts into public.faculty_users (user_id → faculty_id mapping).
       Both functions are SECURITY DEFINER with search_path locked to public.

4.3  Function: public.is_faculty_email(text) → boolean
     - Pure lookup: returns true if email exists in public.directory.
     - Used in the users_insert_self RLS policy to validate faculty role claims at insert.

5. CRITICAL FIELD CONTRACTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

These fields must stay in sync between API route handlers and the DB schema.
Any change to one side requires a corresponding update to the other.

  marketplace.owner_email
    → Must be included in every marketplace INSERT payload.
    → Not derived from auth at read time — stored denormalized.
    → If the user's email changes, existing listings are not auto-updated.

  complaint_upvotes.user_id
    → Must equal auth.uid() at insert time (enforced by RLS).
    → Client must never allow a different user_id to be passed.

  complaint_box.is_anonymous
    → Boolean, DB default is true.
    → Must be explicitly passed in the insert payload to allow non-anonymous posts.
    → Do not rely on the DB default from the client — always send the value.

  Never invoke npx tsc or other .bin tools directly in such paths.
